核心服务层 tss-main/tcs

核心服务层 (Trusted Core Services)，主要实现各种核心功能

tcs/tdd

cmd_man.c

comm_driver.c(提供mmap和ioctl接口的驱动)

这个驱动注册了一个名为 /dev/tpcm_comm 的字符设备（Misc Device）。 它的主要功能是：

  • 提供 mmap 接口：允许用户态进程（Proxy）将内核(cmd_man.c)分配的 vmalloc 内存映射到自己的虚拟地址空间，也就是通过mmap访问这分配的内存。
  • 提供 ioctl 接口：允许用户态进程直接读写内核内存（这通常用于调试或低速通信）。
  • 单例保护：确保同一时间只有一个进程（Proxy）能打开这个设备。

tpcm_comm_mmap函数主要就是为了将内核事先分配好的做共享内存传输数据的那部分内存映射到用户的内存上，也就是将通信双方的逻辑内存映射到相同物理内存上。这样的话用户或内核在这个内存中写数据另一方都能直接看到。

httcsec_miscdev_ioctl函数中用户读写都需要先通过内核的memcpy将数据先拷贝到内核的缓冲区中，之后再拷贝到用户的缓冲区，这是因为用户态的 req.buffer可能是非法地址、可能在 copy 的过程中被换页、被回收、被并发修改、可能指向只读/不可访问区域，如果发生错误，而这一操作又在内核，则内核会直接故障。

这里写的httcsec_miscdev_ioctl函数非常危险，因为没有任何地址合法性检查，没有 access_ok() 检查用户指针，length 也没有限制。并且命令不匹配时也成功返回 0，如果 cmd 的 nr 不是 1 或 2，req_data 还是 NULL，kfree(NULL)没问题，然后return 0;

netlink.c

该代码实现了一个基于 miscdevice 的字符设备，设备名为 tpcm_comm，在用户态对应 /dev/tpcm_comm。该设备提供三类能力：

  1. 通过 open/release 控制设备的独占访问
  2. 通过 mmap 将一段内核 vmalloc 内存映射到用户态
  3. 通过 ioctl 提供内存读写接口
